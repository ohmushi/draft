# Audit du projet Draft — 12 mai 2026

État du projet à date. Analyse multi-axes, du code à l'expérience utilisateur.

---

## Résumé exécutif

Le projet est **bien posé architecturalement** et tient ses promesses fonctionnelles sur les flux principaux. L'architecture Clean est réellement appliquée, les tests couvrent les couches critiques, et l'interface colle au mockup de référence. Les lacunes majeures sont : l'**absence totale d'authentification** (toutes les API sont publiques), des **problèmes de performance** qui s'aggraveront avec le volume, et quelques **violations des règles du projet lui-même** (SVG inline, nav active hardcodée).

---

## 1. Qualité du code

### Forces

- TypeScript strict partout. Aucun `any` (ESLint le bloque en erreur).
- `readonly` systématique sur les types du domaine (`Entry`, `NewEntry`, `CreateEntryInput`).
- `Result<T, E>` implémenté et utilisé dans les use cases (`getEntry`, `deleteEntry`).
- Nommage explicite, conventions respectées (`isEntryTag`, `createEntry`, `EntryNotFoundError`).
- Aucune `!` (non-null assertion) dans le code applicatif.
- Clock injectée dans `createEntry` pour les tests déterministes.

### Problèmes

**SVG inline dans `(main)/layout.tsx`** — L'icône crayon du lien studio et l'icône `+` du FAB sont des SVG inline directement dans le JSX. La règle du projet (`components/svg/`) et l'instruction `architecture.md` sont explicites : tous les SVG doivent être isolés dans `components/svg/`. Les icônes micro et caméra existent déjà comme composants — l'icône crayon et le `+` auraient dû suivre le même chemin.

**SVG inline dans `studio/page.tsx`** — Les icônes play/pause dans les vignettes audio du studio sont également inline, alors que `PlayIcon` et `PauseIcon` existent dans `components/svg/`.

**`nav a.active` hardcodé dans `layout.tsx`** — Le lien "flux" a toujours `className="active"` quelle que soit la page courante. Sur `/about`, la nav affiche `→ flux` comme si on était sur le flux. Next.js App Router expose `usePathname()` pour corriger ça, mais comme le layout est un Server Component, il faudrait soit un petit composant client, soit passer par le pattern `Link` avec `pathname`.

**`mdxComponents` défini inline dans `(main)/page.tsx`** — L'objet `mdxComponents` est recréé à chaque render. Pour un Server Component `force-dynamic`, ce n'est pas un re-render React, mais c'est une habitude à éviter — l'objet devrait être hoissé au niveau module.

**`generateMediaFilename` utilise `Math.random()`** — Pour une collision sur des uploads concurrents, la probabilité est faible sur un projet personnel, mais `crypto.randomUUID()` est disponible nativement et serait plus propre.

**Incohérence `--ink-soft`** — `globals.css` définit `--ink-soft: #cd830a` (orangé) comme dans `mockup.html`. Mais `CONTEXT.md` documente `--ink-soft: #3D3A35` (gris foncé), valeur reprise dans `draft-pwa.html`. La documentation de référence diverge du code réel.

## 2. Architecture

### Forces

- Les couches Clean Architecture sont respectées : `domain/` → `application/` → `infrastructure/` → `app/`.
- `domain/` n'importe rien d'externe. `EntryRepository` est une interface pure.
- Les use cases (`createEntry`, `listEntries`, `getEntry`, `deleteEntry`) reçoivent leurs dépendances en paramètre.
- `InMemoryEntryRepository` et `InMemoryMediaStorage` permettent les tests sans infrastructure.
- `PrismaEntryRepository` est la seule implémentation de production — swappable sans toucher les use cases.
- Séparation `MediaStorage` / `EntryRepository` propre : deux domaines distincts, deux interfaces.

### Problèmes

**Pas de composition root** — Chaque route API et chaque page instancie directement `new PrismaEntryRepository()` et `new MinioMediaStorage(...)`. C'est fonctionnel et acceptable pour un projet solo, mais il n'existe pas de point central où les dépendances sont câblées. Si le repository change de classe, il faut modifier chaque route.

**MDX compilé à chaque requête sans cache** — `(main)/page.tsx` appelle `Promise.all(entries.map(entry => compileMDX(...)))` sur toutes les entrées à chaque requête (`force-dynamic`). Il n'y a aucune mise en cache du MDX compilé par entrée. Avec 50 entrées, chaque visite du flux lance 50 compilations MDX.

**Couplage données / composant dans le contenu MDX** — `buildContent()` produit du MDX avec `<PhotoGrid urls='[...]' />`. Le nom du composant et le nom de la prop `urls` sont encodés dans les données persistées en base. Si `PhotoGrid` est renommé ou si la prop change, toutes les entrées existantes deviennent invalides. C'est une forme de couplage invisible entre la couche application et la couche présentation.

**`PhotoGridCompact` défini inline dans `page.tsx`** — Ce composant wrappeur existe uniquement dans `page.tsx`. S'il est utile dans d'autres contextes (future page de tag, recherche), il faudra le redéfinir. Il mériterait son propre fichier.

## 3. Sécurité

### Criticité haute

**Toutes les API sont publiques et sans authentification.**

- `POST /api/entry` — n'importe qui peut créer des entrées dans le flux.
- `DELETE /api/entry/[slug]` — n'importe qui peut supprimer n'importe quelle entrée.
- `GET /api/media?object=...` — n'importe qui peut accéder à tous les médias stockés dans MinIO.

C'est documenté dans `todo.md` comme une dette connue. Mais en l'état, dès que le site est exposé publiquement, le flux peut être pollué ou effacé sans aucune résistance.

**Path traversal sur `/api/media`** — Le paramètre `object` est passé directement à `minioClient.presignedGetObject(bucket, objectName, ...)`. Si un attaquant envoie `object=../autre-bucket/fichier-sensible`, le comportement dépend de MinIO. Il faudrait valider que `objectName` ne contient pas `..` ni `/` en début.

### Criticité moyenne

**Injection MDX** — Le champ `text` de l'entrée est intégré tel quel dans le MDX compilé par `buildContent()`. Le MDX permet du HTML brut. Un contenu comme `<script>alert(1)</script>` passé dans `text` serait inséré dans le MDX. `next-mdx-remote/rsc` compile vers React, qui échappe le texte, mais pas forcément le HTML brut en MDX. À vérifier et à mitiger (sanitisation du texte avant insertion dans le MDX).

**Pas de limite sur la longueur du texte** — `validateMediaFiles` vérifie les tailles des fichiers mais pas la longueur du champ `text`. Une requête avec un texte de 10 Mo est acceptée.

**Fuite d'informations d'erreur** — Les routes API retournent `error instanceof Error ? error.message : 'Erreur interne'`. Des messages d'erreur Prisma ou MinIO (qui peuvent contenir des noms de tables, des chemins, des adresses internes) peuvent être exposés au client.

**Pas de protection CSRF** — Les endpoints acceptent des requêtes `multipart/form-data` sans token CSRF. Pour `/studio`, dans un contexte authentifié futur, cela pourrait permettre des requêtes forgées depuis un autre site.

### Criticité faible

**Pas de rate limiting** — Les endpoints de création et de suppression d'entrées sont illimités en fréquence.

## 6. Interface et expérience utilisateur

### Points forts

- Le design respecte le mockup de référence : crème, encre, typographies, séparateurs ondulés.
- Les animations `fadeUp` staggerées donnent du rythme au chargement du flux.
- Le studio est immédiatement utilisable : focus automatique sur le textarea, bouton publier désactivé jusqu'à avoir du contenu.
- La sélection de tag toggle (cliquer à nouveau désélectionne) — comportement correct et attendu.
- L'overlay de succès (✓ + "posté dans le flux") est discret et s'efface seul après 1.6s.
- La barre d'enregistrement avec le point clignotant communique l'état de capture de manière claire.
- La vignette audio dans le studio permet d'écouter avant de publier — supprime le risque de publier une mauvaise prise.
- Le FAB `+` est repositionné sur grand écran pour s'aligner avec la colonne de contenu.

### Problèmes

**Lien `/about` visible mais cassé** — C'est visible immédiatement dans la nav. Sur un site présenté à quelqu'un d'autre, ça crée une impression d'inachevé.

**Erreur microphone silencieuse** — Si l'utilisateur refuse la permission micro (ou si `getUserMedia` n'est pas disponible), le `catch` dans `handleToggleRecording` n'affiche rien. L'utilisateur clique sur le bouton, il ne se passe rien, et il ne sait pas pourquoi.

**Erreur de publication générique** — En cas d'échec du `POST /api/entry`, l'interface affiche "erreur — réessayer" sans plus de détail (c'est documenté dans le todo.md). L'utilisateur ne sait pas si c'est un problème réseau, un problème de taille de fichier, ou autre.

**Pas de feedback "en cours" explicite pour l'upload** — Le bouton "publier" passe à "…" pendant la requête, ce qui est suffisant pour du texte. Mais si l'utilisateur poste 5 photos + un audio depuis une connexion mobile, l'upload peut prendre plusieurs secondes sans indicateur de progression.

**Navigation sans état actif** — La nav affiche toujours `→ flux` comme actif. Naviguer vers `/about` ou `/entry/[slug]` ne met pas à jour l'indicateur.

**Pas de métadonnées SEO par entrée** — Toutes les pages partagent `<title>Draft</title>`. Un partage d'une entrée individuelle montre "Draft" sans aucune information sur le contenu.

## 7. Performance

### Problème principal : rendu O(n) du flux

`(main)/page.tsx` exécute `Promise.all(entries.map(e => compileMDX(e.content)))` pour **toutes les entrées** à chaque visite. Avec `force-dynamic`, il n'y a pas de cache de page. À 50 entrées, chaque visite du flux déclenche 50 compilations MDX + 1 requête base de données. La compilation MDX n'est pas gratuite.

Solution naturelle : pagination (10 entrées par page) + cache React (`cache()`) ou `unstable_cache` par slug.

### Médias proxiés

Chaque affichage d'image ou lecture audio passe par `GET /api/media`, qui va chercher une URL signée MinIO puis proxie la réponse complète. C'est intentionnel pour masquer l'endpoint MinIO, mais ça signifie que le serveur Next.js devient un proxy de streaming pour tous les médias. Sur un serveur perso avec peu de RAM, des flux audio concurrents peuvent saturer la mémoire.

`Cache-Control: private, max-age=60` est positionné, ce qui limite les rechargements fréquents — mais 60 secondes reste court pour des médias immuables.

### Pas de `next/image`

Les images sont rendues avec `<img>` natif (via `EntryImage`). Pas de conversion WebP, pas de redimensionnement automatique, pas de lazy loading natif Browser avec priorité LCP. Pour des photos prises sur iPhone (5–10 Mo), ça peut peser lourd sur le flux.

## 8. Maintenabilité et dette technique

### MDX dans la base de données

Stocker du MDX en base (incluant les invocations de composants React comme `<PhotoGrid urls='...' />`) crée une dépendance forte entre les données et le code. Changer le nom ou l'API d'un composant invalide silencieusement toutes les entrées existantes. Il n'y a pas de couche de migration de contenu.

C'est un choix architectural documenté et assumé, mais il mérite d'être anticipé : si `PhotoGrid` évolue, il faudra une migration SQL sur le champ `content`.

### Absence de CI/CD vérifiée

Le dossier `.github/` existe mais son contenu n'a pas été audité. Si les workflows GitHub Actions ne lancent pas les tests et le lint à chaque push, les régressions peuvent s'accumuler silencieusement.

### Tailwind présent mais non utilisé

`postcss.config.mjs` inclut `@tailwindcss/postcss` et `tailwindcss` est dans les devDependencies. Mais le code CSS utilise exclusivement `globals.css` avec des variables CSS custom — aucune classe utilitaire Tailwind n'est employée. La dépendance est un bruit dans le projet qui contredit le principe "pas de sur-ingénierie".
